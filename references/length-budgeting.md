# 篇幅预算

这份 reference 用于把“篇幅目标”从软描述升级成可执行的硬预算。

目标不是单纯把报告写长，而是防止完整报告在长上下文中被压缩成“看起来完成、实际未写满”的状态。

## 第一原则

对 `medium` 和 `full` 交付物，只声明 `short / medium / full` 不够。

必须同时声明：

1. 总字数预算
2. 章节字数预算
3. 达标与未达标的处理规则

## 默认计数字口径

默认使用：`中文正文字数`

计入：

- 正文段落
- 项目符号中的正文
- 表格单元格中的正文
- 数字、年份、产品名等正文内容

不计入：

- markdown 标题符号
- 列表符号
- run 状态说明
- 文件占位标题
- 模板里的空字段提示

这是一种研究写作场景下的实用口径，目标是约束完成度，不是追求字符统计学精度。

## 什么时候必须启用硬预算

以下任一命中，就必须写预算：

1. `medium` 或 `full` 报告
2. `polished` 或 `final` 完成度目标
3. 启用 orchestration 的报告
4. 任何你预期会出现 4 个及以上核心章节的公司深度或行业专题

以下场景可以只保留软篇幅目标：

1. `short` memo
2. 单次 review note
3. 极短摘要或章节 opener 改写

## 必填字段

在 `00_request.md` 里至少写：

- 篇幅档位：`short / medium / full`
- 完成度目标：`rough / working / polished / final`
- 总字数预算：`min / target / max`
- 协作模式：`single-agent / orchestrated-proposed / orchestrated-approved / orchestrated-sequential`

如果是 `medium` 或 `full`，还必须补：

- 章节预算表：每章 `min / target / max`

## 默认总字数预算

如果用户没有指定，使用以下默认值。

### short

- `1500 / 2200 / 3000`

### medium

- `2800 / 3800 / 5000`

### full

- `4500 / 6000 / 8000`

这些是默认值，不是永远正确的真理。若 benchmark、受众或任务形态明显要求更短或更长，可以覆盖，但必须在 `00_request.md` 里显式写出。

## 完整公司深度报告的默认章节预算

当任务是标准公司深度报告，且没有更合适的 benchmark 约束时，优先使用下面这套默认预算。

| 章节 | min | target | max |
| --- | --- | --- | --- |
| 首页核心结论 | 450 | 650 | 850 |
| 公司概况 | 700 | 900 | 1200 |
| 行业/竞争 | 900 | 1200 | 1600 |
| 亮点/差异化 | 900 | 1200 | 1600 |
| 商业化/财务/估值边界 | 1000 | 1400 | 1900 |
| 风险 | 350 | 500 | 700 |

## medium 报告的默认章节预算

| 章节 | min | target | max |
| --- | --- | --- | --- |
| 首页核心结论 | 300 | 450 | 650 |
| 公司概况 | 500 | 650 | 900 |
| 行业/竞争 | 550 | 750 | 1000 |
| 亮点/差异化 | 550 | 750 | 1000 |
| 商业化/财务/估值边界 | 650 | 900 | 1200 |
| 风险 | 200 | 300 | 450 |

## 预算规则

### min

硬下限。

如果总字数或关键章节低于 `min`，默认判定为“未完成”，不能直接进入 polished/final。

### target

默认目标。

如果明显低于 target，优先检查：

1. 是否有章节偷懒
2. 是否 evidence map 没展开
3. 是否本该 research 的部分被一句话带过

### max

压缩上限。

如果明显高于 `max`，优先压缩：

1. 重复背景
2. 章节间同义判断
3. 与 thesis 无关的泛行业铺垫

不要通过砍掉风险或商业化边界来“达标压缩”。

## Budget audit

在 merge 前后都做一次预算审计。

至少检查：

1. 总字数是否达到总 `min`
2. 每个核心章节是否达到本章 `min`
3. 是否存在“前半篇详细、后半篇提纲化”现象
4. 是否有章节明显超配，挤压了更重要章节预算

## 预算不达标时怎么处理

### 情况 1：证据足够，但正文没写开

直接回到章节重写或补写，不进入句子级 polish。

### 情况 2：证据不足，导致写不到预算

先做 targeted research，或收窄本章问题，不要靠行业废话凑字数。

### 情况 3：单章超长、全稿失衡

先重分配章节预算，再压缩重复表述。

## Orchestrated 模式下的特殊规则

1. 总控 agent 先分配章节预算，再分派章节任务。
2. 每个章节 agent 只对本章 `min / target / max` 负责。
3. 合稿前，任何低于本章 `min` 的章节都要先补写。
4. `08_final.md` 必须通过整稿预算审计。

## 反偷懒信号

出现以下迹象时，要优先怀疑报告在偷懒，而不是假设“已经写完”：

1. 前页和公司概况较完整，但后续章节只剩概念性判断
2. 风险章节明显短于预算，且只是空泛免责声明
3. 商业化章节低于预算，且没有展开证据梯度
4. 整体总字数接近下限，但核心章节之间严重失衡

这些都应在 review 中被明确记录，而不是被“文风还不错”掩盖。
