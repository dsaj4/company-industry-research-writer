# Industry / competition 子章节拆解

## 章节目标

- 章节问题：为什么这家公司既有窗口期，也面对更高平台化门槛？
- 章节判断：赛道仍在快速变化，入口心智和产品声量给了 Moonshot 一个窗口，但竞争压力已经从“有没有好产品”升级为“能否把入口沉淀成平台、生态与商业化能力”。
- 章节预算：`min 4665 / target 5131 / max 5597`
- 章节证据包：`B14 B17` + DeepSeek / 豆包 / MiniMax 官方页 + 已有研究结论

## 拆解结构

### 1. 先定义赛道位置，而不是直接讲竞争对手

- 要回答的问题：Moonshot 现在到底处在什么竞争阶段？
- 这一小节先把 AI 原生应用、模型服务、开放平台和 Agent 扩展四层关系说清，避免把“用户热度”误写成“平台壁垒”。
- 需要用的证据：
  - `B14` QuestMobile 口径的 AI 原生应用头部梯队
  - `B17` Similarweb 的 AI referral 流量信号
  - Moonshot / Kimi 官方页与开放平台页

### 2. 给出最小 peer map

- 要回答的问题：谁在分别压 Moonshot 的不同层？
- peer map 必须写清三条：
  - DeepSeek = 开源推理与 Agent 工程压力
  - 豆包 = 巨头分发与企业平台压力
  - MiniMax = 多模态产品矩阵与 Agent 化压力
- 这一小节不展开行业全景，只做最小对照，服务后面的 thesis。

### 3. 拆解三类竞争机制

- 要回答的问题：为什么这些对手不是一般背景噪音，而是平台化门槛？
- 需要分别解释：
  - 开源推理会压缩模型能力溢价
  - 巨头分发会抬高入口和企业触达门槛
  - 多模态矩阵会把单点入口竞争变成系统战
- 重点是“机制”，不是罗列对手名字。

### 4. 解释“窗口期”从哪里来

- 要回答的问题：既然压力这么大，为什么 Moonshot 仍然存在窗口？
- 需要从三个角度写：
  - 用户心智和入口位置已经建立
  - 产品形态和平台能力开始联动
  - 海外 API、开放平台和开发者生态让公司不只是单一 C 端应用

### 5. 收束成 thesis

- 要回答的问题：本章如何服务全篇主判断？
- 结论必须落到一句话：
  - 这不是“赛道很热”的背景章，而是说明 Moonshot 的上限取决于平台化验证，而竞争已经把验证门槛显著抬高。
- 写作边界：
  - QuestMobile 和 Similarweb 只能作为口径化信号
  - 不写绝对性能领先
  - 不把竞争写成泛行业背景
